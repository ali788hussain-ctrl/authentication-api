# Function 1

```python
def hash_password(password):
```

Suppose

User types

```text
Ali123
```

Parameter

```python
password="Ali123"
```

comes here.

---

First line

```python
password.encode()
```

Why?

bcrypt works only with bytes.

Normal string

```text
Ali123
```

becomes

```text
b'Ali123'
```

---

Next

```python
bcrypt.gensalt()
```

What is Salt?

Suppose

Password

```text
Ali123
```

Without salt

Hash always becomes

```text
abc123
```

Hackers can guess it.

Salt adds randomness.

Example

Today

```text
Ali123

↓

Random Salt

↓

Hash

↓

8d91f...
```

Tomorrow

```text
Ali123

↓

Different Salt

↓

Hash

↓

29dkf...
```

Same password

Different hash.

Very secure.

---

Then

```python
bcrypt.hashpw()
```

This combines

Password

*

Salt

↓

Creates

Huge encrypted string

Example

```text
$2b$12$Dkhfjd8738...
```

---

Finally

```python
decode()
```

Remember

bcrypt returns bytes.

MongoDB stores strings.

So

```python
decode()
```

changes

```text
b'$2b$12$...'
```

into

```text
$2b$12$...
```

Now save in database.

---

Whole Flow

```text
Ali123

↓

encode()

↓

Bytes

↓

gensalt()

↓

Random Salt

↓

hashpw()

↓

Encrypted Password

↓

decode()

↓

MongoDB
```

# Function 2

```python
verify_password()
```

Most beginners think

```python
hash(input)==database
```

Wrong.

Because

Hash changes every time.

Instead

bcrypt internally extracts the salt from the stored hash and hashes the entered password with that same salt.

Example

Database

```text
Ali123

↓

$2b$12$ABCXYZ....
```

Login

User types

```text
Ali123
```

Now

```python
bcrypt.checkpw()
```

does

```text
Entered Password

↓

Uses Salt Hidden Inside Stored Hash

↓

Creates Hash Again

↓

Compare

↓

True
```

You never manually compare hashes.

bcrypt handles everything.

# How to Add an Admin?

There are three common ways.

## Option 1 (Recommended for Learning)

Open **MongoDB Compass**.

Find the user document:

```json
{
  "name": "Ali",
  "email": "ali@gmail.com",
  "password": "$2b$12$...",
  "role": "user"
}
```

Edit it to:

```json
{
  "name": "Ali",
  "email": "ali@gmail.com",
  "password": "$2b$12$...",
  "role": "admin"
}
```

Save the document.

Now **log in again**.

Why again?

Because the JWT token stores the role at login time. Your old token still says `"role": "user"`.

After logging in again, the new token will contain:

```json
{
    "email": "ali@gmail.com",
    "role": "admin"
}
```

Now `/admin` will work.

---

## Option 2 (Good for Real Projects)

Allow the registration endpoint to assign roles only under controlled conditions (for example, only an existing admin can create another admin). Normal users should always be created with:

```python
"role": "user"
```

This prevents anyone from registering themselves as an admin.

---

## Option 3 (Production)

Create the very first admin manually using a script or directly in the database. After that, only admins can promote other users.
